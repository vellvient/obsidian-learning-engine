# 01 — Architecture

The system is five cooperating parts. Each is useful alone; together they close the
Bloom two-sigma loop (mastery learning + immediate feedback) for a self-learner.

## The five-part stack

1. **Knowledge graph** — one Obsidian note per skill; lettered sub-skill checkboxes
   are the unit of mastery and SRS; typed prerequisite edges stored in frontmatter
   *and* as body wikilinks (frontmatter for the scripts, wikilinks for the graph view).
2. **Flow-zone diagnostic** (`flow_diagnostic.py`) — computes, from checkbox state +
   edges: SRS due, flow zone (all hard prereqs known, skill not yet mastered), entry
   points, in-progress, blocked, and per-topic mastery bars.
3. **FSRS v6 + chain FIRe** (`srs_fsrs.py`) — spaced repetition scheduling with
   fractional implicit review flowing up the prerequisite chains.
4. **CCT micro-schemas** (`micro_trainer/`) — timed pattern drills for automaticity
   (the HOW-FAST layer next to the graph's HOW/WHY layer).
5. **Daily wrappers** (`morning.py` / `evening.py`) — one command each to open and
   close a study day.

## Design principles (what makes the graph high quality)

These principles come from the reference build and from studying production systems
(Math Academy's ~3,000-topic graph, the open Marble Skill Taxonomy):

1. **Granularity is everything.** Nodes are individual skills ("solve a linear
   inequality with brackets"), not chapters. Sub-skills (`339a`, `339b`, …) are the
   atoms of mastery and review.
2. **The graph must be a DAG.** No cycles, ever. Direction: `from` = prerequisite,
   `to` = dependent.
3. **Typed edges.** `HARD_PREREQ` gates the flow zone; `SOFT_PREREQ` is helpful but
   not gating; `IMPLICIT_REVIEW` marks "practicing the dependent rehearses this
   prerequisite" and powers FIRe.
4. **Transitive reduction.** Never keep A→C when A→B→C exists — redundant edges
   dilute unlock-priority and FIRe computations.
5. **Cross-domain edges matter most.** Algebra inside geometry, stats inside
   mechanics: the edges naive per-chapter mining misses are the valuable ones.
6. **Every edge carries a one-sentence reason.** Unreviewable edges rot.
7. **Mastery lives on the node** (`not-started → attempted → familiar → proficient →
   mastered`) and drives everything downstream.

## Key derived quantities

- **Flow zone** — skills whose HARD prerequisites are all proficient/mastered but the
  skill itself is <100%. This is Math Academy's "just-right difficulty" frontier.
- **Unlock leverage** — how many blocked skills this node is the *sole* remaining gate
  of (`FULLY_UNBLOCKS`), plus in-degree as a hub measure. `study_today.py` ranks
  flow-zone picks by this.
- **Implicit-review credit (FIRe)** — grading a skill boosts the FSRS stability of its
  ancestors by chain weight (see `03-engine-internals.md`).

## Data stores (all plain text/JSON)

| Store | Owner | Notes |
|---|---|---|
| Note frontmatter (`mastery`, `prerequisites`, `leads-to`, `implicit_review`) | scripts + you | The primary database *is the vault* |
| `.engine/prerequisite_edges.json` | mining pipeline | Canonical edge list `[{from, to, reason, type?}]` |
| `.obsidian/srs_state.json` | SRS engine/watcher | Review cards + FSRS sub-objects. **Never hand-edit.** Gitignore it in your own fork if your vault repo is public — it is your personal study history. |
| `.obsidian/graph.json` | you (Obsidian rewrites it) | Color rules. Only edit with Obsidian **closed** |
| `micro_trainer/progress.json` | trainer | Drill fluency stats |

## Mastery synchronization (the 4-place rule)

A note's mastery level is stored in **four** places that must stay in sync, because
different consumers read different ones:

| Place | Consumer |
|---|---|
| `mastery:` frontmatter | scripts |
| `tags:` frontmatter | Obsidian tag pane |
| `#mastered` body tag | Obsidian **graph coloring** |
| `Mastery: **...**` display text | you, reading the note |

`flow_diagnostic.py --sync-mastery` (also run inside `evening.py`) repairs all four
from checkbox truth. Never set mastery by hand without reading the checkboxes; 100%
checkboxes alone caps at `proficient` — promotion to `mastered` needs retrieval proof
(Good/Easy grades in FSRS history).

## Hard constraints (learned the painful way)

- **Never** hand-edit or hand-commit `.obsidian/srs_state.json` — the engine owns it.
- **Never** regenerate `graph.json` while Obsidian is open (it overwrites your file
  within seconds — see `07-pitfalls.md`).
- Auto-generated notes (`00 - Flow Zone Diagnostic`, `00 - Review Grader`,
  `00 - SRS Review Tracker`, and all `practice/*.md`) are regenerated by scripts —
  never hand-edit them.
- Scripts print ASCII-safely (`sys.stdout.reconfigure(errors="replace")`) because
  Windows consoles are often not UTF-8.

## What "auto-generated" means here

The system is idempotent by design: dashboards are regenerated from state, practice
notes are regenerated from the question bank, and inserted blocks in hand-written
notes use HTML marker comments (`<!-- auto:practice-link -->…<!-- /auto:practice-link -->`)
so regeneration replaces exactly its own block and nothing else. Re-running any
script is always safe.
